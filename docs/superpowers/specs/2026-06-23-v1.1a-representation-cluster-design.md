# v1.1a Representation Cluster — Design Spec

**Status:** Approved (brainstorm 2026-06-23)
**Branch:** `feat/v1.1a-representation` (off `main` @ `f9a8f71`)
**Supersedes work in:** open PR #11 `feat/command-expansion` (digested, not merged — see §2)

## 1. Goal

Add five agent commands that surface more of Mol\*'s live interaction through the
existing `Command → adapter → executor → ExecutorContext` seam:
`set-representation`, `set-color`, `toggle-visibility`, `measure-distance`,
`add-label`. The agent-side half is largely **grafted from open PR #11** (clean,
well-described, error codes already in the union); the **new work is the GPU-side
adapter** — the real Mol\* implementations of the new port members, which PR #11
never wrote (it predates Plan 3a's `src/mol/adapter.ts`).

This is **v1.1a** — the first of two v1.1 sub-clusters. `load-scene`, `toggle-xr`,
and `highlight.style` are **v1.1b** (separate spec), because each carries a larger
open question and `toggle-xr` cannot even be GPU-verified without a headset.

## 2. Why PR #11 is digested, not merged

PR #11 (`feat/command-expansion`, opened 2026-06-20) branched from `50a2579` —
**before** Plan 3a, the trajectory cluster, and packaging. Merging it as-is is
blocked by three facts, verified firsthand 2026-06-23:

1. **8 files conflict** (`merge-tree` "changed in both"): `commands.ts`,
   `context.ts`, `executor.ts`, `types.ts`, two test files, plus
   `wiki/command-schema.md` / `wiki/index.md` — all moved by 3a / trajectory /
   packaging.
2. **`raw/0009` collision** — PR #11 claims `raw/0009`, but `main`'s `0009` is
   `plan3a` (next free id is `0014`).
3. **Fatal semantic gap** — PR #11 adds four `ExecutorContext` members
   (`setRepresentation`/`setColor`/`setVisibility`/`addLabel`) but never
   implements them in `src/mol/adapter.ts` (which did not exist when PR #11 was
   written). Merging it would leave `molstarExecutorContext` not satisfying the
   `ExecutorContext` interface → `pnpm typecheck` fails. **The GPU-side adapter
   half is exactly what PR #11 lacks, and exactly the meat of this cluster.**

So PR #11 is treated as **design input + ~80%-reusable agent-side code**. Its
types/commands/executor cases/`measure.ts`/tests are cherry-picked into a fresh
branch; the adapter implementations are written new. After this cluster merges,
PR #11 is closed with a note (see §8).

## 3. Command set (the contract)

All five reuse the existing LLM-friendly `Selection` type and the existing
`ErrorCode` union — **no new error codes** (`invalid_input`, `no_structure`,
`empty_selection`, `internal_error`, `unknown_command` already cover them).

| command | input | result | Mol\* mapping |
|---|---|---|---|
| `set-representation` | `{ selection, type }` | `ok()` | loci→component, set the component's representation `type` |
| `set-color` | `{ selection, scheme? , color? }` (exactly one) | `ok()` | loci→component, `updateRepresentationsTheme` (base color) |
| `toggle-visibility` | `{ selection, visible }` | `ok()` | loci→component, `setSubtreeVisibility` |
| `measure-distance` | `{ from, to }` | `ok({ distanceAngstrom })` | **pure-Node** centroid distance — no port member |
| `add-label` | `{ selection, text }` | `ok()` | `measurement.addLabel(loci, { visualParams: { customText } })` |

### Enums (in `src/types.ts`, molstar-free)

- `REPRESENTATION_TYPES = ['cartoon', 'ball-and-stick', 'spacefill',
  'molecular-surface', 'gaussian-surface', 'point', 'line', 'ellipsoid']`
- `COLOR_SCHEMES = ['element', 'chain', 'residue-index', 'secondary-structure',
  'b-factor', 'hydrophobicity', 'sequence-id']`

These are the agent-facing names; the adapter maps them to Mol\*'s internal
representation-type and color-theme registry names (see §5, spike 2).

### Naming decision (locked)

Adopt PR #11's `set-*` / `toggle-*` scheme verbatim: `set-representation`,
`set-color` (not the older `color` sketch), `toggle-visibility`,
`measure-distance`, `add-label`. Consistent with the existing `set-frame`.

### `set-color` validation (locked, from PR #11)

`scheme` XOR `color` — enforced by the **executor**, not the JSON Schema
(JSON Schema can't cleanly express "exactly one"). `color` must match
`/^#[0-9a-fA-F]{6}$/`. Validate the color spec **before** resolving the loci, so a
malformed color fails fast.

## 4. Architecture

Two halves, split by the molstar-free boundary (unchanged from the project's
existing seam):

```
agent-side (molstar-free, src/index.ts barrel)      browser-side (molstar, src/browser.ts)
  types.ts   (enums + input types)        ─────►      executor.ts  (5 new dispatch cases)
  commands.ts (5 new CommandSpec)                       │ resolveSelection → loci
                                                        ▼
                                              context.ts  ExecutorContext + 4 new members + ColorSpec
                                                        │
                                              ┌─────────┴──────────┐
                                       mol/adapter.ts          test fake
                                    (real molstar impl)     (records calls)
                                              │
                                       measure.ts (pure-Node centroid/distance, executor-only)
```

### Component model (locked): replace-in-place

`set-representation` / `set-color` / `toggle-visibility` target a **per-selection
visual component**. Repeated styling of the *same* selection updates that one
component in place (so "show chain A as cartoon" then "as spacefill" **replaces**,
preserving any color already set) — it does not stack independent
representations. The adapter derives a **stable component key from the loci
itself** (its units + element ranges), so PR #11's `(loci, …)` port signatures
stay unchanged — the executor still passes only a loci. Components are cached in a
`Map<key, ref>` on the adapter instance; `plugin.clear()` in `loadStructure`
already wipes them on a new load (confirm — spike 4).

## 5. File-by-file changes

### Agent-side (must stay molstar-free — guarded by `assert-agent-side-molstar-free.mjs`)

**`src/types.ts`** — add (pure TS, no molstar import):
- `REPRESENTATION_TYPES` / `RepresentationType`, `COLOR_SCHEMES` / `ColorScheme`
  (`as const` arrays + derived unions, matching the existing preset/enum pattern).
- `SetRepresentationInput`, `SetColorInput`, `ToggleVisibilityInput`,
  `MeasureDistanceInput`, `MeasureDistanceResult`, `AddLabelInput` interfaces
  (verbatim from PR #11).

**`src/commands.ts`** — add the five `CommandSpec` entries with PR #11's
LLM-facing descriptions and JSON Schemas. Reuse the existing shared
`selectionSchema` fragment. **Do not** reintroduce PR #11's `highlight.style`
schema — `highlight` stays `{ selection }` (style is v1.1b). `measure-distance`'s
schema uses `from` / `to` (two `selectionSchema` copies).

### Browser-side

**`src/executor.ts`** — graft five `case` blocks into the existing dispatch
switch (which already has the trajectory cases). Add the small validators from
PR #11 that aren't already present: `requireBoolean`, `requireEnum`,
`requireString`, `requireColorSpec`, and the `nonEmptyLociFor` helper
(resolve-then-reject-empty). `requireColorSpec` returns a `ColorSpec`. Reuse the
existing `asObject` / `requireSelectionAt` / `lociFor` helpers and the
`ExecutorError` → `CommandResult` mapping. Routing/validation mirrors PR #11's
executor verbatim, adapted to main's current helper names.

**`src/measure.ts`** (new) — `centroidOfLoci(loci): Vec3` and
`distanceBetweenLoci(a, b): number`, pure data-model (molstar `Vec3` +
`StructureElement.Loci.forEachLocation`), no plugin/WebGL — Node-unit-testable
like `resolveSelection`. Imported only by the executor (browser-side); never
reaches `src/index.ts`. Verbatim from PR #11.

**`src/context.ts`** — add the `ColorSpec` type (`{ scheme: ColorScheme } | { hex:
string }`) and four members to `ExecutorContext`:
```ts
setRepresentation(loci: StructureElement.Loci, type: RepresentationType): void;
setColor(loci: StructureElement.Loci, color: ColorSpec): void;
setVisibility(loci: StructureElement.Loci, visible: boolean): void;
addLabel(loci: StructureElement.Loci, text: string): void;
```

### The new work — `src/mol/adapter.ts`

Implement the four members against Mol\* 5.10.1. The shared primitive is a
**loci→component bridge**:

- `componentFor(loci): StructureComponentRef` (or its representation ref) —
  derive a stable key from the loci, return the cached component if present, else
  create one scoped to the loci and cache it. Likely path:
  `StructureElement.Bundle.fromLoci(loci)` → a `StructureSelectionQuery` →
  `plugin.builders.structure.tryCreateComponentFromSelection(structure, query,
  key)`. (Exact API is spike 1.)

Then:
- `setRepresentation(loci, type)` — bridge → set that component's representation
  to the mapped Mol\* type, replacing its prior representation type while keeping
  any color theme already applied.
- `setColor(loci, color)` — bridge → `plugin.managers.structure.component
  .updateRepresentationsTheme([comp], { color: 'uniform', colorParams: { value:
  Color(0xRRGGBB) } })` for a hex, or `{ color: <mapped scheme name> }` for a
  scheme. **Base color** via `updateRepresentationsTheme`, **not** `applyTheme`
  (which is overpaint/transparency overlay — wiki [[molstar-api]]).
- `setVisibility(loci, visible)` — bridge → `setSubtreeVisibility(plugin.state
  .data, comp.cell.transform.ref, !visible)`.
- `addLabel(loci, text)` — `plugin.managers.structure.measurement.addLabel(loci,
  { visualParams: { customText: text } })`. No bridge needed — the measurement
  manager consumes a loci directly. (`customText` confirmed present on the
  `StructureSelectionsLabel3D` visual params in 5.10.1.)

`measure-distance` needs **no** adapter member — the executor calls the pure-Node
`distanceBetweenLoci` directly and returns `ok({ distanceAngstrom })`.

### Test fake

The in-test fake `ExecutorContext` (used by the Node executor tests) gains the
four new members as call-recorders (push args to arrays), so the executor's
routing/validation can be asserted without a GPU.

## 6. Testing strategy (follows the existing off-GPU / GPU split)

**Automated (Node, in CI):**
- `test/measure.test.ts` — `centroidOfLoci` / `distanceBetweenLoci` on Node-built
  fixtures (known coordinates → known distance). From PR #11.
- `test/executor.test.ts` additions — for each of the five commands: a
  well-formed `Command` routes to the right fake-port member with the right args;
  malformed inputs (bad enum, both/neither of scheme|color, non-hex color,
  missing text, empty selection, no structure) return the right `error.code`.
  From PR #11, adapted to main's helpers.
- `test/commands.test.ts` additions — the five specs are present, have the
  expected `name`/`required`/enum shapes, and derive into Anthropic tools via the
  existing `toTools`.

**Manual (GPU demo, not CI):** a new `RepresentationPanel` in `examples/demo/`
(Plan 3b style) driving the five commands on a real loaded structure:
representation change (cycle the types), recolor by scheme and by hex,
hide/show a selection, distance readout between two selections, and a placed
text label. The adapter + the loci→component bridge are **typecheck-gated +
GPU-verified by hand** (no unit test exercises a real WebGL context), matching
how Plan 3a/3b/trajectory adapters are verified.

## 7. Implementation risks (resolve during planning — the plan carries real code)

1. **loci→component bridge** — the exact `Bundle.fromLoci` → `StructureSelectionQuery`
   → `tryCreateComponentFromSelection` call and the component-key derivation.
   This is the central new helper; spike it first against 5.10.1 `.d.ts`.
2. **Name mappings** — agent enum → Mol\* registry names for representation types
   (e.g. `ball-and-stick`, `gaussian-surface`) and color schemes (e.g.
   `element` → `element-symbol`, `chain` → `chain-id`). Verify each against the
   representation/color-theme registries; an unmapped name must fail cleanly, not
   silently no-op.
3. **Color persistence across a representation change** — confirm in the demo that
   `set-representation` after `set-color` keeps the color (the replace-in-place
   model intends it to).
4. **Component cleanup on reload** — confirm `plugin.clear()` in `loadStructure`
   evicts the cached components (and clear the adapter's `Map` there too).

## 8. Out of scope (this spec)

- `load-scene` (MolViewSpec `loadMVS`), `toggle-xr`, `highlight.style` → **v1.1b**.
- Label removal/restyling (place text only), per-atom / nearest-atom distance
  (centroid-only, as PR #11), angle/dihedral measurement, representation opacity,
  multi-structure scoping.
