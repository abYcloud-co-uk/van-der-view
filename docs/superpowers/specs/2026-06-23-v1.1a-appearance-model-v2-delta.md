# v1.1a Appearance Model — final (per-selection component), GPU-informed

**Status:** Implemented, awaiting GPU re-verification (2026-06-23)
**Amends:** `2026-06-23-v1.1a-representation-cluster-design.md` §4 (component model) + §5 (adapter)
**Supersedes:** the v2 overpaint/transparency draft (below, "Rejected approaches")
**Branch:** `feat/v1.1a-representation` (PR #21, draft)

## Why two rewrites

The original v1 adapter layered a per-selection component **on top of** the `default`
preset → double-draws and a stray whole-chain ball-and-stick. The v2 rework moved
color onto the preset (overpaint for hex, structure-wide `updateRepresentationsTheme`
for schemes) and hid the preset under per-selection transparency. A second external
review + a GPU pass proved v2 was *also* architecturally broken: color lived in a
different place (the preset) than the representation it was supposed to color (the vdv
component), so the two never composed.

### What the GPU pass actually showed (the evidence for this model)

| test | observation | conclusion |
|---|---|---|
| A: 1CRN → spacefill | clean CPK spacefill, **no cartoon ghost** | **transparency-hide of the preset works** — no component-subtraction / carve-out needed |
| B: color blue → spacefill | spacefill reverts to CPK | color did **not** persist across a representation change (v2 never stored it on the entry) |
| C: spacefill → color blue | spacefill turns blue | overpaint reached the visible draw only because `presetComponents()` happened to include the vdv comp — fragile, single-selection-only |
| D: hex blue → b-factor | stays solid blue | a stale overpaint masked the scheme — color-on-preset and theme-on-preset fight |
| E: 1HSG → b-factor | whole structure recolored | structure-wide `updateRepresentationsTheme` ignores the selection |

The decisive positive result is **A**: transparency-hiding the preset is visually correct,
so the fix does **not** need Mol*'s component subtraction. The fix is simply to put **all**
appearance on the per-selection component and hide the preset's coverage of it.

## The final model — one owned component per selection

Each selection touched by `set-representation` / `set-color` / `toggle-visibility` gets
**one vdv component** (created via `tryCreateComponentFromExpression`, keyed by a
full-identity `lociKey`). That component owns its representation **and** its color; the
preset's draw of those atoms is hidden with per-loci transparency so there is no
double-draw. No structure-wide mutation, no clear-all, no overpaint.

- **`set-representation(loci, type)`** — get-or-create the component, `applyStyle(type, entry.color)` (rebuild the single representation, **carrying the stored color**), then hide the preset coverage (see below).
- **`set-color(loci, color)`** — get-or-create the component, `applyStyle(entry.reprType ?? defaultReprFor(loci), color)`, then hide the preset coverage. Color is applied to the component's representation: **hex** → `color: 'uniform' + colorParams.value`; **scheme** → `color: <theme name>`, which Mol\* scopes to the component's atoms → **schemes are now per-selection**. Color is stored on the entry, so it survives later representation changes.
  - `defaultReprFor(loci)`: polymer → `cartoon`, else → `ball-and-stick` (mirrors the default preset; no more dense whole-chain ball-and-stick).
- **Hiding the preset coverage** (`hidePresetCoverage(loci, type)`) — `setStructureTransparency(presetOnly, 0, () => loci)` then `setStructureTransparency(presetOnly, 1, () => toHide)`, where `toHide` is the **polymer subset** of the selection for a `cartoon` (which draws only polymer) and the **whole selection** otherwise. Restoring the whole selection first means switching rep types never leaves stale-hidden atoms; hiding only what the rep draws means a `{chain:'A'}` color keeps chain A's waters/ligands visible (drawn by the preset) instead of vanishing — the GPU pass caught them disappearing under a naïve full-selection hide. A single selection still draws with one representation, so a deliberately mixed `set-representation cartoon` over a ligand renders nothing for that ligand (acceptable). Transparency layers accumulate across repeated styling (Mol\* appends; `Transparency.merge` collapses at render) — fine for v1.1a interaction volumes.
- **`toggle-visibility(loci, visible)`** — if the selection owns a component, toggle it with `setSubtreeVisibility` (the preset is already hidden). Otherwise set per-loci transparency `visible ? 0 : 1`; a later `0`-layer shadows the prior `1`-layer for the same atoms (`Transparency.merge` processes latest-first), so restore is scoped — other selections are untouched.
- **`add-label(loci, text)`** — unchanged: `measurement.addLabel(loci, { visualParams: { customText } })`, deleting the prior label tracked for that loci-key (replace-in-place).
- **`measure-distance`** — unchanged (pure-Node, no port member).

### Why this fixes each finding

- **color persistence (#1)** — color is stored on the entry and re-applied by `applyStyle` on every representation change.
- **color lands on the visible draw (#2)** — color is the component's own representation color; there is no separate preset draw to land on.
- **no cross-selection wipe (#3/#4)** — every mutation is per-component / per-loci; nothing clears all overpaint or all transparency.
- **per-selection schemes (#5)** — a color theme on a sub-structure component colors only that component's atoms.
- **hex ↔ scheme don't fight (#6)** — both go through the one `applyStyle` path that rebuilds the component's representation; there is no overpaint layer left to mask a theme.
- **collision-free key (#7)** — `lociKey` is the full element-index list per unit, not bounds+size.
- **#9** — `residue-index` and `sequence-id` both map to Mol\*'s `sequence-id` (Mol\* has no distinct residue-index theme); documented equivalence.
- **async port (#1 of the first review)** — unchanged from v2: the four mutators are `Promise<void>` and the executor `await`s them, so a GPU throw becomes a reported `internal_error`.

## Port / cache / testing

- **Port** (`src/context.ts`): `setRepresentation` / `setColor` / `setVisibility` / `addLabel` are `(…) => Promise<void>`; the executor `await`s each. (Unchanged from v2.)
- **Cache** (`components: Map<lociKey, CompEntry>`): per selection tracks `component`, `repr`, `reprType`, `color`, `label`. `components.clear()` on reload.
- **Serialization**: all four mutators run on a single global op chain (`serialize`). They share plugin state — the one preset transparency cell and the component tree — so even calls on *different* selections must not interleave their read-modify-write commits (a per-loci-key lock would still let a chain-A vs chain-B pair race on the shared transparency cell). The ops are infrequent GPU writes, so a global chain is free.
- **Testing**: the executor + port behavior is Node-unit-tested (routing + throw→`internal_error` + the `null`-as-absent color guard); the adapter is typecheck-gated and **GPU-verified by the user** (color persistence, per-selection schemes on 1HSG, cross-selection independence, per-loci hide/show).

## Known limitations (single-representation model)

A selection owns **one** component drawn with **one** representation. That keeps the model simple
and composable, at these costs:

- **Mixed polymer+ligand selections** — a selection spanning a chain *and* a bound ligand draws as one type. `set-color`'s default (`cartoon`, since the chain is polymer) doesn't render the ligand, and the preset-hide leaves the ligand drawn by the preset in its *original* color → the command reports success but the ligand keeps the preset color. (Waters/ions in the selection stay visible, just uncolored — the GPU-round-1 fix.) The proper fix is multi-representation components (cartoon for polymer + ball-and-stick for non-polymer, both colored) — deferred to v1.1b.
- **Visibility ordering on mixed selections** — `set-visibility` toggles the vdv component once a selection is styled, vs. per-loci preset transparency before. For a *mixed* selection, hide → color → re-hide can leave its non-polymer atoms visible (the second hide only toggles the vdv component, which draws polymer). Pure-polymer / single-kind selections are unaffected. Same root as above (one component), same v1.1b fix.
- **Single structure** — the transparency `lociGetter` returns the bound loci and ignores the `structure` argument Mol\* passes, which is correct only while there is one loaded structure (the executor reads `structures[0]`). Multi-model/assembly components or a structure rebuilt by a frame update would silently no-op the hide. Revisit when multi-structure lands.
- **`residue-index` ≡ `sequence-id`** — Mol\* has no distinct residue-index color theme, so both schemes map to `sequence-id` (rainbow N→C) and render identically. Kept as agent-friendly synonyms; documented so it isn't mistaken for a bug.
- **Transparency layer growth** — repeated styling appends transparency layers (Mol\* collapses them at render via `Transparency.merge`, but the stored list grows). Fine for interaction volumes.

## Out of scope (still)

`load-scene` / `toggle-xr` / `highlight.style` (v1.1b). Per-sub-selection multi-representation styling (v1.1b).

## Rejected approaches (kept for the record)

- **v1 (per-selection component over the preset)** — double-draws; toggle-visibility doesn't hide because the preset still draws the atoms.
- **v2 (color on the preset: hex→overpaint, scheme→structure-wide retheme; preset hidden under transparency)** — color and representation live in different places and never compose: color doesn't persist across representation changes, hex masks schemes, schemes are structure-wide, and the clear-all idempotency wipes other selections. Disproven on GPU (table above).
